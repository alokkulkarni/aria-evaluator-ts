# Project Overview

## What is ARIA Evaluator?

**ARIA Evaluator** is a TypeScript-first evaluation platform for testing conversational AI agents used in contact-centre and banking-style support journeys. It can drive live conversations against multiple backends, capture transcripts and audio, score outcomes with an LLM-as-judge, and surface the results through both a CLI and a browser UI.

The repository is not just an app server. It contains:

- the **evaluation runtime** (`src/`)
- the **React operator UI** (`src/ui/`)
- **provider adapters** for Amazon Connect, Lex, Azure Direct Line, Strands/AgentCore, OpenAPI HTTP endpoints, and WebSocket bots
- **infrastructure code** for local Docker, AWS ECS/Fargate, CloudFront, and an optional Bedrock proxy Lambda
- **scenario libraries**, generated transcripts/reports, and fine-tuning utilities

## Core Purpose

The project is built to answer a practical question: **"How well does an AI agent handle realistic and adversarial customer interactions?"**

It does that by combining four capabilities:

1. **Scenario execution** against chat and voice providers
2. **Customer simulation** using either scripted turns or a Bedrock-driven customer persona
3. **Evaluation** with dimension-based quality and security scoring
4. **Artifact generation** through transcripts, HTML/JSON reports, logs, and optional voice recordings

## Key Dependencies

- **`@aws-sdk/client-bedrock-runtime`** for customer simulation and LLM judging
- **`@aws-sdk/client-connect` / `@aws-sdk/client-connectparticipant`** for Amazon Connect chat and voice setup
- **`@aws-sdk/client-lex-runtime-v2`** for Lex chat evaluation
- **`@deepgram/sdk`** for custom voice provider flows
- **`@prisma/client` / `prisma`** for persistence
- **`react` / `react-dom` / `vite`** for the UI
- **`ws`** for WebSocket chat and voice adapters

## Key Features

- **Multi-provider evaluation** across AWS-native and generic HTTP/WebSocket agents
- **Chat and voice execution**, including WebRTC-based Amazon Connect voice runs
- **Scenario-driven testing** using YAML files, multi-document scenario files, and a scenario builder UI
- **Security and adversarial testing** for prompt injection, system prompt exfiltration, authority impersonation, and related attacks
- **LLM-based scoring** with separate quality and security dimensions
- **Live run streaming** to the UI via server-sent events
- **Report generation** as JSON and styled HTML
- **Dual deployment paths** for local Docker and AWS-hosted environments

## Non-source and Supporting Content

- **`scenarios/`** contains curated scenario suites in categories such as `banking`, `escalation`, `edge_cases`, and `adversarial`
- **`data/finetune/`** and `scripts/*finetune*` support distilling judge outputs into Bedrock fine-tuning datasets
- **`data/`, `reports/`, and `transcripts/`** hold runtime artifacts already present in the checkout
- **`dist/`** is compiled output from `npm run build`

## Getting Started

### Local application development

```bash
npm ci
npm run dev
```

This starts:

- the **API server** on port `3001`
- the **Vite UI** on port `5173`
- local proxying from UI to API for `/api`, `/reports`, `/transcripts`, and `/audio`

### Production-style local build

```bash
npm run build
npm start
```

### CLI execution

Representative commands:

```bash
npm run cli:connect
npm run cli:lex
npm run cli:azure
npm run cli:openapi
```

### Local Docker mode

```bash
docker compose up --build
```

Optional local Bedrock proxy:

```bash
docker compose --profile bedrock up --build
```

### AWS deployment

There are two infrastructure paths:

- **Terraform** under `infra/terraform/`
- **CloudFormation low-cost flow** via `scripts/deploy-ecs-cloudfront.sh`

## Build and Validation Surface

The repository currently exposes these main quality gates in `package.json`:

- **`npm run lint`** → TypeScript type-check (`tsc --noEmit`)
- **`npm run build`** → API compilation + Vite UI build

There is **no dedicated test script** in `package.json` at the time of analysis; validation is centered on type-checking, building, and real run execution.

## Architecture Summary

ARIA Evaluator is best understood as a **modular monolith with a worker-style execution path**:

- the **Express API** handles operator interactions, stores metadata, and serves artifacts
- the **CLI runtime** executes actual scenario runs and writes transcripts/reports to disk
- the **conversation engine** coordinates scripted or agent-driven customer turns
- the **adapter layer** talks to whichever provider is under evaluation
- the **judge** scores outcomes after the run completes

A notable design choice is that **filesystem artifacts are first-class runtime outputs**. Transcripts, logs, audio, and reports are produced on disk first, then the API ingests and indexes them into SQLite so the UI can browse them efficiently. That makes the system work both as a local developer tool and as a deployable evaluation service.
